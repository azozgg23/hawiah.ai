High severity

F1
Dead except StorageException sends provider errors to 500 instead of a classified 502
Highcorrectness · regression confidence: highverified ✓ backend/app/routers/generations.py:339-371, 459-478

This branch changed the inner exception handler around the provider call from except Exception to except StorageException. The inner try wraps only the openai_generate / gemini_generate call — those raise httpx.HTTPStatusError (OpenAI) and google.genai SDK errors (Gemini), never storage3's StorageException. The handler is therefore dead code, so classify_provider_error() is never reached for provider failures.

Evidence — the inner try wraps the provider call only
try:
    if body.provider is ProviderEnum.openai:
        result = await asyncio.wait_for(openai_generate(...), timeout=120.0)
    else:
        result = await asyncio.wait_for(asyncio.to_thread(gemini_generate, ...), timeout=120.0)
except asyncio.TimeoutError as exc:
    raise ProviderError("TIMEOUT", ...) from exc
except ProviderError:
    raise
except StorageException as exc:           # <-- never raised here; classify_provider_error is dead
    code, user_msg = classify_provider_error(exc)
    raise ProviderError(code, user_msg) from exc

A raw provider error (401 invalid key, 429 rate-limited, 400 content policy, network drop) now slips past this block and is caught by the outer except Exception, which marks the row failed with a hard-coded INTERNAL_ERROR and returns HTTP 500:

except Exception:
    logger.exception(...)
    _mark_failed(generation_id, "INTERNAL_ERROR", "Internal error")
    raise _error_response(500, "INTERNAL_ERROR", "Something went wrong. Please try again.")
Cross-file confirmation (provider sources read during verification)

openai_image.py:46 calls response.raise_for_status(), which raises a raw httpx.HTTPStatusError — never pre-wrapped — and classify_provider_error() (error_mapping.py:22-45) maps those httpx errors to INVALID_KEY (401/403), RATE_LIMITED (429), CONTENT_POLICY (400/422 policy), and PROVIDER_CLIENT/SERVER_ERROR. Every one of those now regresses to 500. The only survivors are the two genuine ProviderError paths: EMPTY_RESPONSE (raised by both providers, caught by except ProviderError: raise) and the asyncio.wait_for TIMEOUT (caught by except asyncio.TimeoutError). Gemini SDK errors aren't httpx either, so they hit the same dead branch and regress from the old coarse PROVIDER_SERVER_ERROR (502) to INTERNAL_ERROR (500).

Impact
Every provider-originated failure now returns 500 INTERNAL_ERROR with a generic message instead of 502 + the actionable code (INVALID_KEY, RATE_LIMITED, CONTENT_POLICY, NETWORK). The DB row is still marked failed, but with the wrong error_code. Users can no longer tell a bad API key from a transient outage — and the "Review your provider keys" link (see F3) loses its trigger. These two bugs compound: together they completely break the invalid-key recovery flow.
Suggested direction
Restore except Exception as exc for the provider-call block. The genuine StorageException surface is the watermark download() and the upload() calls lower in the function (lines 379-389), which are currently unwrapped — wrap those if storage errors should be classified.
F2
Detail endpoint returns 500 for pending / processing generations (response-model enum mismatch)
Highcontract-mismatch confidence: highverified ✓ models/generation.py:78-80, 105-121 · routers/generations.py:518-563

Found independently by two lanes (backend + contract). GenerationDetailResponse.status is typed as GenerationHistoryStatusEnum, which accepts only succeeded / failed. But the DB column (generation_status_t, migration 00006) has four values — pending, processing, succeeded, failed — and the detail lookup applies no status filter:

def _find_generation_or_404(client, brand_id, generation_id):
    result = (client.table("generations").select("*")
        .eq("id", str(generation_id))
        .eq("brand_id", str(brand_id))
        .maybe_single().execute())          # no .in_("status", ["succeeded","failed"])
    ...
# models/generation.py
class GenerationHistoryStatusEnum(str, Enum):
    succeeded = "succeeded"
    failed = "failed"
class GenerationDetailResponse(BaseModel):
    status: GenerationHistoryStatusEnum     # <-- rejects pending/processing
Impact
Fetching the detail of a generation in pending/processing state raises a Pydantic ValidationError at response-serialization time → unhandled HTTP 500. Reachable during the live ~120 s generation window, and permanently for any row stuck in processing after a crash/restart mid-generation (the generate handler is blocking, so an interrupted run never transitions to failed).

Reachability caveat: the history list query filters to succeeded/failed, so a user cannot reach this by normal list → click navigation. It triggers only via a direct/bookmarked/shared detail URL or a crash-stuck row — i.e. off the happy path. Rated High because it is an unhandled 500 on a valid, owned resource that becomes permanent for stuck rows; if you weight reachability heavily, Medium is also defensible.
Suggested direction
Either add .in_("status", ["succeeded","failed"]) to the detail lookup (404 for non-terminal rows), or widen GenerationDetailResponse.status to the full GenerationStatusEnum and have the frontend render an in-progress state. (Delete reaches the same helper but never serializes status, so it is unaffected.)
F3
Backend error code is discarded in api.ts — the INVALID_KEY remediation link can never render
Highcorrectness confidence: highverified ✓ lib/api.ts:4-77 · hooks/use-generate.ts:31-36 · generation/error-message.tsx:15

The backend error envelope is { error: { code, message, request_id } }. getErrorMessage() extracts only message, and apiRequest throws a plain new Error(message) — the code is dropped on the floor. Downstream, use-generate reads (err as {code?}).code (always undefined on a plain Error) and falls back to scraping the message for an ALL-CAPS token:

// use-generate.ts
let code = (err as { code?: string }).code              // undefined
if (!code && err instanceof Error) {
  const match = err.message.match(/\b([A-Z][A-Z0-9_]{1,})\b/)
  if (match) code = match[1]
}
code = code ?? 'UNKNOWN'

The INVALID_KEY user message — "Your provider key was rejected. Please check your keys." — contains no run of ≥2 capitals, so the regex never matches and code resolves to 'UNKNOWN'. error-message.tsx only renders the recovery link when code === 'INVALID_KEY':

{code === 'INVALID_KEY' && (
  <Link href={`/${brandId}/keys`}>Review your provider keys</Link>
)}
Impact
The "Review your provider keys" remediation link is dead for every invalid-key failure. Combined with F1 (which already downgrades INVALID_KEY to INTERNAL_ERROR server-side), the entire invalid-key recovery UX is broken end-to-end. The brittle regex also mis-classifies any message that happens to contain an unrelated acronym.
Suggested direction
Throw a typed ApiError extends Error carrying code from errorPayload.error.code; read err.code directly in use-generate and delete the regex fallback.

Medium severity

F4
DELETE removes storage before the DB row — a partial failure leaves a dangling row with a dead image URL
Mediumdata-integrity confidence: mediumverified ✓ backend/app/routers/generations.py:592-610

The delete handler removes the storage object first, then deletes the DB row with no error handling on the second call:

image_path = row.get("image_path")
if image_path:
    try:
        client.storage.from_(settings.STORAGE_BUCKET).remove([image_path])   # 1) storage gone
    except StorageException as exc:
        if _is_already_missing_storage_error(exc): pass
        else: raise _error_response(502, "STORAGE_DELETE_FAILED", ...)

client.table("generations").delete().eq("id", str(generation_id)) \
    .eq("brand_id", str(brand_id)).execute()                                  # 2) if THIS throws, row survives
Impact
If the DB delete throws after storage removal succeeds, the row survives with an image_path pointing at a deleted object. History then shows a broken image, and the detail page hands back a valid-looking image_url that 404s. Recoverable on retry (storage-missing is tolerated), but the interim state is corrupt.
Suggested direction
Delete the DB row first, then best-effort delete storage (log & swallow). Worst case becomes an orphaned PNG with no DB reference — a harmless storage leak — instead of a visible broken record.
F5
Post-delete refetch discards all previously loaded pages when the cursor boundary shifts
Mediumcorrectness · UX confidence: highverified ✓ frontend/hooks/use-generation-history.ts:78-135

After a successful delete the page calls refetch(), which runs loadPages(activeCursor). To "restore" scroll depth, it walks pages forward looking for a live next_cursor equal to the stored activeCursor. Because cursors are keyset cursors over (created_at, id), deleting any item at or before that boundary shifts every subsequent boundary, so the stored cursor will never re-appear. The walk falls through to the replace branch:

if (!cursorFound && next !== cursorToRestore) {
  const targetPage = await requestPage(cursorToRestore)
  loadedItems = targetPage.items     // <-- REPLACES; pages 1..N-1 thrown away
  next = targetPage.next_cursor
}

The same branch fires unconditionally once the user has loaded more than 10 pages (the guard < 10 cap stops the forward walk early).

Impact
Deleting an item after using "Load more" collapses the visible list to a single 24-item page; items the user was looking at vanish. Recoverable only by a manual full refresh.
Suggested direction
On delete, refetch() from a clean page-1 (activeCursor = null) and drop the cursor-reconstruction path, or optimistically splice the deleted id out of items instead of re-walking.
F6
Generate proxy route has no try/catch — timeout/network errors surface as an opaque 500 HTML page
Mediumerror-handling confidence: highverified ✓ frontend/app/api/brands/[brandId]/generate/route.ts:21-29

The proxy's fetch to the backend uses AbortSignal.timeout(290_000) but is wrapped in no try/catch. If the backend is unreachable or the 290 s timeout fires, the route handler throws and Next.js returns a 500 with an HTML body:

const upstream = await fetch(`${BACKEND_URL}/brands/${params.brandId}/generate`, {
  method: 'POST', headers, body,
  signal: AbortSignal.timeout(290_000),     // throws AbortError -- uncaught
})
const responseBody = await upstream.text()

Client-side, api.ts only parses JSON when content-type says so; on an HTML 500 it falls back to response.statusText → the user sees a bare "Internal Server Error".

Impact
A slow or down backend yields an opaque error with no timeout/connectivity signal — indistinguishable from a real server bug, and it bypasses the structured error UI entirely.
Suggested direction
Wrap the fetch/text() in try/catch; return 504 on AbortError and 502 on other network errors, both with a JSON { error: { code, message } } body the client can parse.

Low severity

F7
Download failures silently swallowed — try/finally with no catch
Lowerror-handling confidence: highverified ✓ components/history/history-download-button.tsx · components/generation/generator-result.tsx

Both download buttons call downloadImageFile() inside try { … } finally { setDownloading(false) } with no catch. downloadImageFile throws on any non-2xx fetch (expired/missing object, CORS): the rejection goes unhandled, the button flips back to "Download", and the user gets no error feedback — they can't tell success from failure.

Suggested direction
Add a catch that surfaces the thrown message (already descriptive — "Failed to fetch image") via local state or a toast.
F8
URL.revokeObjectURL called synchronously after click() — flaky download on Firefox/Safari
Lowcorrectness · cross-browser confidence: mediumverified ✓ frontend/lib/download.ts:10-12 
a.click()
a.remove()
URL.revokeObjectURL(url)   // revoked before the browser has read the blob

This shared helper (new on this branch, now used by both the generator result and history download) revokes the object URL in the same tick as the programmatic click. Chrome tolerates it; Firefox/Safari may cancel the download, yielding a 0-byte or failed file.

Suggested direction
Defer revocation: setTimeout(() => URL.revokeObjectURL(url), 10_000) after the click.
F9
Generation PNGs live in a public bucket — readable by anyone with the URL (pre-existing design)
Lowsecurity · privacy confidence: highverified ✓ supabase/migrations/00009_create_storage_bucket.sql

The brand-assets bucket is public = true with a Public Read policy for the public role, and the API contract intentionally serves /object/public/… URLs. Generation images at brands/{brandId}/generations/{id}.png are therefore fetchable by anyone holding the URL, with no auth check, until the generation is deleted.

This is a deliberate decision from feature 006, not a regression introduced here — included for completeness. If brand assets ever need confidentiality, switch to signed URLs (storage.createSignedUrl) and a private bucket. The history feature's delete is currently the only privacy lever.

F10
Keyset cursor encodes a +00:00 offset into the PostgREST or filter — confirm URL-encoding is intact
Verifycorrectness confidence: lowneeds runtime test backend/app/routers/generations.py:_decode_cursor / list query

The pagination cursor decodes created_at back to an ISO string with a +00:00 offset and interpolates it into a PostgREST filter:

query.or_(f"created_at.lt.{cursor_created_at},"
          f"and(created_at.eq.{cursor_created_at},id.lt.{cursor_id})")
# cursor_created_at looks like 2026-06-07T12:00:00+00:00

A literal + in a URL query string decodes to a space. supabase-py / postgrest-py route this value through httpx, which should percent-encode it to %2B — in which case it's fine. But if any layer passes it through unencoded, PostgREST would read the timestamp as a space-separated value and the boundary comparison would silently shift or fail, causing skipped or duplicated rows at page edges. The keyset logic itself (desc order, id tiebreaker, fetch N+1 for has_next) is otherwise correct.

How to confirm
Paginate a brand with >24 generations whose created_at values straddle a page boundary and assert no item is skipped or repeated across pages; or capture the outbound PostgREST request and verify the + is encoded as %2B.
